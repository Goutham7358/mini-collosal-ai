\begin{abstract}
Training large language models requires distributing computation across multiple GPUs, but production frameworks hide the underlying communication logic behind high-level APIs. We present \textit{Mini Colossal-AI}, a distributed training library built entirely from raw \texttt{torch.distributed} primitives, implementing four parallelism strategies, namely data parallelism (DP), ZeRO optimizer sharding, 1D tensor parallelism (TP), and pipeline parallelism (PP), with multiple variants each, composed through a unified 3D plugin modeled after Colossal-AI's process group mesh. We benchmark standalone and 2-axis hybrid configurations on a 4-GPU all-TCP cluster, then scale to a heterogeneous 8-GPU cluster with full 3D parallelism (DP$\times$PP$\times$TP). Communication-aware placement yields up to 28.5\% higher throughput and is essential for training stability. TP memory savings enable 16$\times$ larger batch sizes, pushing peak throughput to 7,387~tok/s (4.73$\times$ single-GPU speedup). Extending to the T5-base encoder-decoder model, we find that architectural asymmetry reduces scaling at small batches but large batches close the gap.
\end{abstract}

\begin{IEEEkeywords}
Distributed Training, Data Parallelism, Pipeline Parallelism, Tensor Parallelism, ZeRO, Communication-Aware Placement, 3D Hybrid Parallelism
\end{IEEEkeywords}

\section{Introduction}

The growth of Transformer-based models from millions to billions of parameters~\cite{DBLP:journals/corr/abs-1810-04805, DBLP:journals/corr/abs-2005-14165} has created a ``memory wall'' where model parameters, gradients, and optimizer states together exceed the capacity of a single GPU. Distributed training addresses this by spreading computation across multiple devices, but different strategies involve different trade-offs: Data Parallelism (DP) replicates the model on every GPU, Tensor Parallelism (TP)~\cite{DBLP:journals/corr/abs-1909-08053} splits individual layers, Pipeline Parallelism (PP)~\cite{DBLP:journals/corr/abs-1811-06965, 10.1145/3341301.3359646} distributes layers across devices, and ZeRO~\cite{rajbhandari2019zero} partitions optimizer states. Production frameworks such as DeepSpeed~\cite{rasley2020deepspeed} and Colossal-AI~\cite{10.1145/3605573.3605613} implement and compose these strategies, but they hide the underlying communication logic behind high-level APIs, making it difficult to reason about how each strategy works.

To develop this understanding from first principles, we present \textit{Mini Colossal-AI}, a distributed training library built entirely from raw \texttt{torch.distributed} primitives. We implement four parallelism strategies, namely DP (with four gradient sync variants), ZeRO Stage~1 and~2, 1D TP (Megatron-LM style), and PP (naive and 1F1B schedules), and compose them into hybrid configurations through a unified 3D plugin modeled after Colossal-AI's process group mesh~\cite{10.1145/3605573.3605613}.

We benchmark all standalone and 2-axis hybrid configurations on a 4-node all-TCP cluster, then scale to a heterogeneous 8-GPU cluster and activate all three axes simultaneously. We systematically study \textit{communication-aware placement}, which is the mapping of parallelism axes to network tiers, and further extend our framework to T5-base~\cite{raffel2020exploring}, an encoder-decoder architecture, to test whether our findings generalize across fundamentally different model structures.

Our contributions are: (1)~a from-scratch implementation of four parallelism strategies with multiple variants each, composed through a unified 3D plugin. (2)~A systematic placement study showing that correct axis-to-link assignment is not merely a performance optimization but is essential for training stability. (3)~A demonstration that TP-enabled memory savings unlock 16$\times$ larger batch sizes, shifting the system from communication-bound to compute-bound operation. (4)~A multi-architecture comparison showing that pipeline imbalance and cross-attention overhead reduce T5's hybrid scaling at small batch sizes, but large batches amortize (spread out) these overheads and close the gap with GPT-2. (5)~An analysis of ZeRO-1 composition with full 3D parallelism.

\section{Related Work}

\textbf{Data Parallelism (DP)} replicates the model on every GPU and synchronizes gradients after each backward pass. The main bottleneck is the all-reduce communication. Ring all-reduce~\cite{patarasuk2009bandwidth} distributes the reduction across all nodes in $2(P{-}1)$ steps, achieving bandwidth-optimal communication. Gradient bucketing~\cite{li2020pytorch} further improves this by batching many small gradient tensors into larger buffers before triggering all-reduce, reducing the number of communication rounds.

\textbf{ZeRO}~\cite{rajbhandari2019zero} addresses the memory redundancy in DP, where every GPU stores the full optimizer states, gradients, and parameters. Stage~1 partitions only the optimizer states, while Stage~2 also partitions gradients, reducing per-GPU memory from $16P$ bytes to roughly $4P + 12P/N$ or $4P + 4P/N$ respectively, where $P$ is the parameter count and $N$ is the number of GPUs.

\textbf{Tensor Parallelism (TP)} splits individual layers across GPUs. Megatron-LM~\cite{DBLP:journals/corr/abs-1909-08053} introduced 1D TP for transformers, where each attention head and MLP slice is computed on a separate GPU, requiring two all-reduces per transformer block.

\textbf{Pipeline Parallelism (PP)} assigns consecutive layers to different GPUs. GPipe~\cite{DBLP:journals/corr/abs-1811-06965} splits mini-batches into micro-batches for concurrent execution but suffers from pipeline flushes. PipeDream~\cite{10.1145/3341301.3359646} introduced the 1F1B (one-forward-one-backward) schedule, which interleaves forward and backward passes to keep more stages busy simultaneously.

\textbf{Colossal-AI}~\cite{10.1145/3605573.3605613} unifies these strategies through a process group mesh, an $N$-dimensional grid of ranks where each axis corresponds to a parallelism dimension (DP, PP, TP), enabling arbitrary combinations without code duplication. Our work follows this design closely.

\textbf{Communication-Aware Placement.} In 3D hybrid parallelism, which axis maps to which network tier matters significantly. Megatron-LM~\cite{DBLP:journals/corr/abs-1909-08053} keeps TP within NVLink-connected nodes and PP across nodes, since PP only performs point-to-point sends while TP requires frequent all-reduces. Our work validates these placement heuristics empirically, tests three distinct axis-to-link assignments, and demonstrates the stability implications of incorrect placement.

\textbf{T5 Encoder-Decoder Model.} Raffel et al.~\cite{raffel2020exploring} proposed T5, casting every NLP task as sequence-to-sequence generation. Unlike decoder-only GPT-2~\cite{radford2019language}, each T5 decoder block contains an additional cross-attention sublayer attending to the encoder output, increasing both parameter count per block and TP communication volume.

\section{Methodology}

\subsection{Hardware Setup}

Standalone and 2-axis hybrid experiments run on 4~AWS g4dn.xlarge instances (1~T4 GPU each), connected via TCP over a VPC private network. For full 3D hybrid experiments, we use 2~AWS g4dn.12xlarge instances, each with 4~NVIDIA Tesla T4 GPUs (16\,GB VRAM each). Intra-node GPU communication uses PCIe Gen3 (${\sim}15$\,GB/s), and inter-node communication uses TCP over VPC (${\sim}0.6$\,GB/s). This 25$\times$ bandwidth gap between intra-node and inter-node links creates the heterogeneous environment central to our placement study. All experiments use PyTorch~2.4.1 with CUDA 12.1 and the NCCL backend.

\subsection{Models and Data}

All models are implemented from scratch in PyTorch. We use two GPT-2~\cite{radford2019language} configurations:
\begin{itemize}
\item \textbf{GPT-2 Small}: 12~layers, $d{=}768$, 12~heads, ${\sim}117$M parameters. Each block contributes 2~TP all-reduces (one per sub-layer), giving 24 all-reduces per forward pass.
\item \textbf{GPT-2 Medium}: 24~layers, $d{=}1024$, 16~heads, ${\sim}354$M parameters. The 24~identical blocks split evenly across 2~pipeline stages (12 each), producing near-perfect compute balance. 48 all-reduces per forward pass.
\end{itemize}

\textbf{T5-base}~\cite{raffel2020exploring} is an encoder-decoder transformer (${\sim}237$M parameters) with 12~encoder layers ($d{=}768$, 12~heads) and 12~decoder layers. Each decoder block adds a cross-attention sub-layer attending to the encoder output, requiring 3~TP all-reduces per block versus GPT-2's~2. This gives 60 total per forward pass (24~encoder + 36~decoder), 25\% more than GPT-2 Medium. For pipeline parallelism, the encoder maps to stage~0 and the decoder to stage~1, creating an inherent compute imbalance because decoder blocks are heavier.

The dataset is WikiText-2~\cite{merity2016pointer} (${\sim}2.4$M tokens), tokenized with the GPT-2 byte-pair encoding, with a fixed sequence length of 256 tokens and a vocabulary size of 50,257.

\subsection{Data Parallelism}

Each GPU holds a full copy of the model and processes a distinct data shard. After the backward pass, gradients are averaged across all GPUs before the optimizer step. Our implementations use two communication operations from PyTorch's \texttt{torch.distributed} library~\cite{li2020pytorch}: \texttt{dist.send}/\texttt{dist.recv}, where one GPU sends gradients directly to another, and \texttt{dist.all\_reduce}, where all GPUs participate at once to sum their gradients, with NCCL internally choosing the best algorithm (ring, tree, etc.).

We implement four synchronization variants. First, the naive approach uses \texttt{dist.send}/\texttt{dist.recv} in an all-to-all pattern with $O(P)$ bandwidth cost. Second, ring all-reduce~\cite{patarasuk2009bandwidth} arranges GPUs in a ring and performs reduce-scatter and all-gather phases via \texttt{dist.send}/\texttt{dist.recv}, achieving a bandwidth-optimal cost of $2N(P{-}1)/P$. Third, the ring-bucketed variant groups gradients into 25\,MB buckets~\cite{li2020pytorch} before applying the same ring algorithm, reducing communication rounds from one per parameter to roughly five per step. Fourth, the NCCL-bucketed variant uses the same bucketing but calls \texttt{dist.all\_reduce} on each bucket, letting NCCL handle the communication internally. This variant also accepts an optional process group so that gradient averaging can be restricted to a subset of GPUs in hybrid configurations.

\subsection{ZeRO Optimizer Sharding}

We implement ZeRO~\cite{rajbhandari2019zero} Stage~1 and Stage~2 as custom optimizer wrappers. In both stages, the model's parameters are flattened into a single vector and divided equally among $P$ GPUs. Each GPU only creates Adam optimizer states (momentum and variance) for its own $1/P$ partition, cutting optimizer memory by a factor of $P$.

In Stage~1, after the backward pass, gradients are averaged across all GPUs using \texttt{dist.all\_reduce}. Each GPU then runs the Adam update only on its own partition and calls \texttt{dist.all\_gather} so every GPU ends up with the full updated model.

Stage~2 saves additional memory by also partitioning gradients. Each GPU only keeps the gradient slice for its own partition and discards the rest. After updating its partition, each GPU broadcasts its updated parameters to all others using \texttt{dist.broadcast}. This reduces both gradient and optimizer memory by a factor of $P$.

Stage~1 accepts an optional process group parameter, allowing it to run within a subset of GPUs for hybrid parallelism.

\subsection{1D Tensor Parallelism}

Following Megatron-LM~\cite{DBLP:journals/corr/abs-1909-08053}, we implement two custom linear layers: \textbf{ColumnParallelLinear} (splits weights along columns, no forward communication, all-reduce on backward) and \textbf{RowParallelLinear} (splits along rows, all-reduce to sum partial outputs). In each transformer block, the attention Q/K/V projections use column-parallel and the output projection uses row-parallel. The MLP is split similarly. This gives 2 all-reduces per block, totaling 48 per forward pass for our 24-layer GPT-2 Medium model.

\subsection{Pipeline Parallelism with 1F1B}

We partition the model's transformer blocks evenly across pipeline stages and communicate activations (forward) and gradients (backward) via \texttt{dist.send}/\texttt{dist.recv}. We implement two schedules: the \textbf{naive} schedule processes all micro-batches sequentially through each stage with bubble ratio $\approx (S{-}1)/S$, while the \textbf{1F1B} schedule interleaves one forward and one backward pass per stage after a warmup of $S{-}1$ forwards, reducing the bubble to $(S{-}1)/(S{-}1+M)$. With $S=4$ stages and $M=8$ micro-batches, this drops the bubble from 75\% to $\sim$27\%. Both schedules support rank remapping for use in hybrid configurations.

\subsection{Hybrid Parallelism and the Unified Plugin}

Following Colossal-AI's approach~\cite{10.1145/3605573.3605613}, we wrote a single unified plugin that takes TP size, PP size, and ZeRO stage as arguments and computes $\text{dp\_size} = \text{world\_size} / (\text{tp\_size} \times \text{pp\_size})$ automatically. The core idea is a \textbf{3D process group mesh}: all GPUs are arranged in a grid with axes (DP, PP, TP), and \texttt{torch.distributed} sub-groups are created along each axis. Every standalone component accepts an optional process group. The plugin passes the DP sub-group to gradient sync, PP neighbor ranks to the pipeline schedule, and the TP sub-group to column/row-parallel layers.

On the 4-GPU all-TCP cluster, this limits us to two axes at a time (e.g., DP$\times$PP or DP$\times$TP). With 8~GPUs on the heterogeneous cluster, we activate all three axes simultaneously in a $(2,2,2)$ mesh where each GPU belongs to one TP, PP, and DP group. For pipeline stages, each stage wraps its transformer blocks in tensor-parallel versions. Replicated parameters (embeddings, LayerNorm, \texttt{lm\_head}) are synchronized across the TP group before DP gradient synchronization to ensure consistency.

\subsection{Communication-Aware Placement}

With 8~GPUs arranged in a $(2,2,2)$ mesh, each GPU belongs to exactly one TP, PP, and DP sub-group. The three axes have very different communication profiles: DP performs one all-reduce of the full gradient buffer (${\sim}1.4$\,GB for GPT-2 Medium) per step, TP performs many small all-reduces at every transformer block boundary (48 per forward pass for GPT-2 Medium), and PP sends only small point-to-point activation tensors between adjacent stages. On our heterogeneous cluster, the mesh axis ordering determines which axis maps to the slow inter-node TCP link (${\sim}0.6$\,GB/s) versus the fast intra-node PCIe (${\sim}15$\,GB/s). The first (slowest-varying) axis spans across nodes. We test three placements:

\begin{itemize}
\item \textbf{Good}: mesh\,=\,(pp,\,dp,\,tp). PP goes inter-node, while DP and TP stay intra-node.
\item \textbf{Bad}: mesh\,=\,(dp,\,pp,\,tp). DP goes inter-node, while PP and TP stay intra-node.
\item \textbf{Worst}: mesh\,=\,(tp,\,pp,\,dp). TP goes inter-node, while PP and DP stay intra-node.
\end{itemize}

Since PP communicates only small activation tensors, it is the most tolerant of slow links, while the bandwidth-heavy DP and latency-sensitive TP should reside on the fast intra-node interconnect. Fig.~\ref{fig:good_placement} illustrates the good placement: TP pairs (e.g., GPU~0$\leftrightarrow$1) and DP pairs (e.g., GPU~0$\leftrightarrow$2) communicate over PCIe within each node, while PP links (e.g., GPU~0$\leftrightarrow$4) cross the slow inter-node TCP link.

\subsection{Evaluation Metrics}

We report \textit{aggregate throughput} (tokens/second across all GPUs), \textit{peak GPU memory} per pipeline stage, and \textit{speedup} relative to the single-GPU baseline at the same or best-case batch size, across 20--30 training steps per experiment.

\section{Results and Discussion}

\subsection{Standalone Strategies on the 4-GPU TCP Cluster}

We first evaluate each strategy in isolation on the 4-node all-TCP cluster using GPT-2 Medium (354M parameters, bs=4 per GPU unless noted). The single-GPU baseline throughput is 1,361~tok/s with 7.67\,GB peak memory.

Fig.~\ref{fig:pcie_tcp} compares each standalone strategy on both the all-TCP cluster and a single PCIe node. On TCP, PP achieves the highest throughput (3,070~tok/s) because it only performs point-to-point sends between adjacent stages, while DP and TP suffer from frequent all-reduce operations over the slow network. TP provides the best memory reduction (56\%) but its 48 all-reduces per forward pass add ${\sim}316$\,ms of overhead on TCP. Moving to PCIe (${\sim}15$\,GB/s), DP, ZeRO, and TP all see 2.4--3.0$\times$ speedups because they rely on bandwidth-heavy all-reduce or all-gather operations, while PP improves by only 4\% because its small point-to-point activation tensors barely saturate even the TCP link. This asymmetry directly motivates the communication-aware placement study: PP should be assigned to the slow inter-node link while DP and TP benefit from fast intra-node PCIe.

\subsection{Communication-Aware Placement}

Fig.~\ref{fig:placement} shows the impact of placement on GPT-2 Medium throughput at bs=8 with the full $(2,2,2)$ 3D configuration. Good placement (PP inter-node) delivers 4,851~tok/s, which is 14.9\% faster than bad placement and 28.5\% faster than worst placement. PP tolerates the slow TCP link because it sends only small point-to-point activation tensors between adjacent pipeline stages. In contrast, placing DP inter-node forces ${\sim}1.4$\,GB of gradient synchronization through the 0.6\,GB/s TCP bottleneck every step. Placing TP inter-node is the most damaging because TP performs frequent all-reduces at every transformer block boundary, and TCP's high latency (${\sim}$ms round-trip) compounds across 48 operations per forward pass.

\textbf{Stability impact.} Beyond throughput, placement critically affects training stability. Bad and worst placements \textit{deadlock} at batch sizes ${\geq}16$ due to NCCL peer-to-peer contention when large activation tensors compete with DP/TP traffic on the same inter-node TCP link. Good placement avoids this entirely because PP's small point-to-point messages do not cause contention. This means correct placement is not merely a performance optimization. It is \textbf{essential for reliable training at scale}.

This 37\% reduction per GPU enables dramatically larger effective batch sizes. Table~\ref{tab:batch} shows how throughput scales with batch size under good placement.

Throughput increases by 52\% from bs=8 to bs=128, with the steepest gains between bs=8 and bs=32 (+36\%). Even under good placement, at small batch sizes the GPUs are communication-bound: they spend a significant fraction of each step waiting for gradient synchronization and pipeline activation transfers. Beyond bs=64, returns diminish sharply (+2\% from bs=64 to bs=128), indicating the system has become compute-bound. The practical sweet spot is bs=32--64, achieving ${\sim}90$\% of peak throughput while retaining memory headroom for larger models.

\subsection{Pipeline Imbalance Analysis}

T5-base's lower scaling at small batch sizes stems from \textit{pipeline stage imbalance}. Table~\ref{tab:imbalance} shows per-stage memory: GPT-2's identical blocks split into near-perfect halves, while T5-base's encoder-decoder split creates a 62.7\% imbalance because each decoder block contains an additional cross-attention sublayer (${\sim}7$M extra parameters). In the 1F1B schedule, the lighter encoder idles waiting for the slower decoder. A secondary factor is TP communication: T5-base requires 60 all-reduces per forward pass versus 48 for GPT-2 Medium (25\% more), further penalizing T5 at small batch sizes.

\begin{table}[h]

\subsection{Summary of Key Findings}

The main quantitative findings are:

\begin{itemize}
\item \textbf{Good vs.\ worst placement (+28.5\%).} PP tolerates slow links, while TP/DP need fast PCIe.
\item \textbf{Placement stability.} Bad/worst placements deadlock at bs${\geq}$16. Correct placement is essential, not optional.
\item \textbf{Batch scaling (4,851$\rightarrow$7,387 tok/s).} TP memory savings enable 16$\times$ larger batches, shifting the system from communication-bound to compute-bound.
\item \textbf{Architecture effect (3.21$\times$ vs.\ 1.55$\times$).} Symmetric decoder-only models scale better than encoder-decoder models at small batch sizes.
\item \textbf{T5 catch-up (3.49$\times$ vs.\ 3.51$\times$).} Large batches amortize architectural overhead, closing the gap.
\item \textbf{ZeRO-1 overhead ($-$9.2\%).} Manageable cost, most valuable for larger models where optimizer memory is the binding constraint.
\end{itemize}

\section{Conclusion and Future Work}

We presented \textit{Mini Colossal-AI}, a from-scratch distributed training library, and used it to systematically study communication-aware 3D hybrid parallelism on a heterogeneous GPU cluster. Our experiments show that correct placement is essential for both throughput and stability, that TP-enabled memory savings create a virtuous cycle enabling 16$\times$ larger batches (peak 4.73$\times$ speedup), and that model architecture fundamentally shapes scaling, though large batches close the gap between symmetric and asymmetric models. The peak speedup is 4.73$\times$ rather than the theoretical 8$\times$ because pure 8-way data parallelism is infeasible because GPT-2 Medium already requires 11.1\,GB at bs=8, exceeding single-T4 capacity once optimizer states are included, so the hybrid axes are necessary to fit the model at all. Within this hybrid setup, pipeline bubbles waste ${\sim}11$\% of GPU time, TP adds 48 all-reduces per forward pass, and DP gradient synchronization adds further per-step communication cost, together limiting realized speedup to ${\sim}59$\% of ideal.

\textbf{Future work.} Several directions remain. (1)~\textit{Heterogeneous stage splitting}: splitting T5's encoder and decoder into unequal numbers of pipeline stages to reduce imbalance (e.g., 8~encoder layers on stage~0, 4~encoder + 12~decoder layers split across stages~1--2). (2)~\textit{Higher TP degrees}: testing TP=4 within a node to further reduce per-GPU memory and enable even larger models. (3)~\textit{NVLink evaluation}: moving to instances with NVLink (e.g., p3.8xlarge with V100s) to measure how the placement trade-offs change when intra-node bandwidth increases from 15\,GB/s (PCIe) to 300\,GB/s (NVLink). (4)~\textit{Mixed precision training}~\cite{micikevicius2018mixed}: combining FP16 computation with our parallelism strategies to further reduce memory and communication volume. (5)~\textit{Larger models}: testing GPT-2 Large (774M) and T5-large (770M) to evaluate how findings scale beyond the 100--350M parameter range.
